# Coding Rules

These rules apply to all code generated by this platform.

---

## Universal rules

- All code must be production-ready: no `TODO`, `FIXME`, or placeholder implementations.
- No hardcoded secrets, credentials, or environment-specific values in code.
- All public APIs must have error handling.
- Code must pass the project's linter with zero warnings.

---

## Go rules

- Use `gofmt` formatting.
- Error messages must be lowercase and not end with punctuation.
- Exported functions must have godoc comments.
- Use `context.Context` as the first parameter in all service functions.
- Avoid `init()` functions.
- Use `errors.Is` / `errors.As` for error comparison, not string matching.

---

## TypeScript / React rules

- Enable strict TypeScript mode.
- No `any` type unless absolutely unavoidable (must be justified with a comment).
- Use `const` by default; use `let` only when mutation is needed.
- Prefer named exports over default exports for components (exception: page components).
- No direct DOM manipulation. Use React state.

---

## SQL rules

- All queries must be parameterized. No string concatenation in SQL.
- Migrations must be reversible (include `down` migration).
- Use explicit column lists in `INSERT` and `SELECT` statements. Avoid `SELECT *`.

---

## General

- Commit messages follow the Conventional Commits format (see `configs/github.json`).
- One logical change per commit.
- Do not commit commented-out code.
